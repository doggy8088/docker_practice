## 9.7 容器網路高階屬性

深入探討容器網路的核心機制、Overlay 網路、CNI 外掛生態、容器 DNS 解析、網路策略等高階屬性，為生產級別的網路架構打下堅實基礎。

### 9.7.1 Overlay 網路原理與設定

Overlay 網路在現有網路基礎上建立虛擬網路，允許容器跨宿主機通訊。它是 Kubernetes 和 Swarm 模式的基礎。

#### Overlay 網路工作原理

Overlay 網路透過隧道封裝技術（通常是 VXLAN）將容器網路流量封裝在宿主機物理網路的 UDP 資料套件中傳輸。

```
容器 A (192.168.0.2)
    ↓
veth 對
    ↓
br-net (網橋)
    ↓
Docker 引擎 (VXLAN 封裝)
    ↓
物理網路 (172.16.0.0/24)
    ↓
Docker 引擎 (VXLAN 解封裝)
    ↓
br-net (網橋)
    ↓
veth 對
    ↓
容器 B (192.168.0.3，不同宿主機)
```

#### 建立和使用 Overlay 網路

**Docker Swarm 模式下的 Overlay 網路：**

```bash
# 初始化 Swarm（建立叢集）
docker swarm init

# 建立 overlay 網路
docker network create --driver overlay \
  --subnet 192.168.0.0/24 \
  --opt com.docker.network.driver.mtu=1450 \
  my-overlay-net

# 驗證網路建立
docker network ls
docker network inspect my-overlay-net

# 在 Swarm 服務中使用 overlay 網路
docker service create --name web \
  --network my-overlay-net \
  --replicas 3 \
  nginx:latest

# 驗證服務跨節點通訊
docker service ps web
```

**單機 Overlay 網路模擬（Linux 容器）：**

```bash
# 建立自定義 overlay 網路
docker network create --driver overlay custom-overlay

# 建立兩個容器
docker run -d --name container1 --network custom-overlay nginx:latest
docker run -d --name container2 --network custom-overlay nginx:latest

# 測試跨容器通訊
docker exec container1 ping container2
docker exec container1 curl http://container2

# 檢查網路設定
docker network inspect custom-overlay
```

#### Overlay 網路效能最佳化

```bash
# 調整 MTU（Maximum Transmission Unit）避免分片
# VXLAN 開銷 50 位元組，物理 MTU 1500，建議設定為 1450
docker network create --driver overlay \
  --opt com.docker.network.driver.mtu=1450 \
  optimized-overlay

# 啟用 IP 地址管理（IPAM）自定義
docker network create --driver overlay \
  --subnet 10.0.9.0/24 \
  --aux-address "my-router=10.0.9.2" \
  my-custom-overlay

# 在 Compose 中使用 overlay 網路
version: '3.9'
services:
  web:
    image: nginx
    networks:
      - backend

  db:
    image: postgres
    networks:
      - backend

networks:
  backend:
    driver: overlay
    driver_opts:
      com.docker.network.driver.mtu: 1450
```

### 9.7.2 CNI 外掛生態概覽

容器網路介面（CNI）是容器編排平台（尤其是 Kubernetes）的標準化網路介面。不同的 CNI 外掛提供不同的網路能力。

#### 主流 CNI 外掛對比

**Calico - 基於 BGP 的網路**

Calico 使用 BGP 協定進行路由，支援網路策略和 eBPF 加速。

```yaml
# Kubernetes 中安裝 Calico
apiVersion: v1
kind: ConfigMap
metadata:
  name: calico-config
  namespace: kube-system
data:
  cni_network_config: |
    {
      "name": "k8s-pod-network",
      "cniVersion": "0.4.0",
      "plugins": [
        {
          "type": "calico",
          "datastore_type": "kubernetes",
          "mtu": 1450,
          "ipam": {
            "type": "calico-ipam"
          }
        },
        {
          "type": "portmap",
          "snat": true,
          "capabilities": {"portMappings": true}
        }
      ]
    }
```

**Flannel - 簡單可靠的 Overlay**

Flannel 提供簡單的 overlay 網路實現，適合小到中等規模的叢集。

```bash
# 安裝 Flannel
kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml

# 設定 Flannel 後端（VXLAN、UDP、AWS VPC 等）
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: kube-flannel-cfg
  namespace: kube-flannel
data:
  cni-conf.json: |
    {
      "name": "cbr0",
      "cniVersion": "0.3.1",
      "plugins": [
        {
          "type": "flannel",
          "delegate": {
            "hairpinMode": true,
            "isDefaultGateway": true
          }
        },
        {
          "type": "portmap",
          "capabilities": {
            "portMappings": true
          }
        }
      ]
    }
  net-conf.json: |
    {
      "Network": "10.244.0.0/16",
      "Backend": {
        "Type": "vxlan"
      }
    }
EOF
```

**Cilium - eBPF 驅動的先進網路**

Cilium 使用 eBPF 在核心級別實現網路策略和可觀測性，效能優異。

```bash
# 安裝 Cilium
helm repo add cilium https://helm.cilium.io
helm install cilium cilium/cilium \
  --namespace kube-system \
  --set image.tag=v1.14.0 \
  --set operator.replicas=1

# 啟用 Hubble（可觀測性）
helm upgrade cilium cilium/cilium \
  --namespace kube-system \
  --reuse-values \
  --set hubble.enabled=true \
  --set hubble.ui.enabled=true
```

**Weave - 跨主機網路通訊**

Weave 提供簡單的跨主機通訊，支援加密和多播。

```bash
# Docker 中使用 Weave 網路
docker run -d --name weave \
  --net=host \
  --cap-add=NET_ADMIN \
  --cap-add=SYS_PTRACE \
  ghcr.io/weaveworks/weave:latest

# 連線到 Weave 網路
docker run -d --network weave --name web nginx:latest
```

**CNI 外掛對比表：**

| 屬性 | Calico | Flannel | Cilium | Weave |
|------|--------|---------|--------|-------|
| 路由方式 | BGP | VXLAN/UDP | eBPF | VxLAN |
| 網路策略 | ✓ | ✗ | ✓ (L3-L7) | ✗ |
| 效能 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 可觀測性 | 中 | 低 | ⭐⭐⭐⭐⭐ | 中 |
| 學習曲線 | 中 | 低 | 高 | 低 |
| 生產就緒 | ✓ | ✓ | ✓ | ✓ |

### 9.7.3 容器 DNS 解析機制

Docker 內建 DNS 伺服器，但 DNS 解析涉及多個層面的設定。

#### DNS 解析流程

```
容器應用 (dig www.example.com)
    ↓
容器內 /etc/resolv.conf (127.0.0.11:53)
    ↓
Docker 內嵌 DNS 伺服器 (127.0.0.11)
    ↓
使用者自定義 DNS 或宿主機 /etc/resolv.conf
    ↓
外部 DNS 伺服器 (8.8.8.8 等)
    ↓
DNS 回應 → 容器快取 → 應用
```

#### 設定容器 DNS

**在執行時指定 DNS：**

```bash
# 單個容器
docker run -d \
  --dns 8.8.8.8 \
  --dns 1.1.1.1 \
  --dns-search example.com \
  nginx:latest

# DNS 選項
docker run -d \
  --dns-option ndots:2 \
  --dns-option timeout:1 \
  --dns-option attempts:3 \
  nginx:latest

# 檢視容器 DNS 設定
docker exec <container_id> cat /etc/resolv.conf
```

**Docker Compose DNS 設定：**

```yaml
version: '3.9'

services:
  web:
    image: nginx
    dns:
      - 8.8.8.8
      - 1.1.1.1
    dns_search:
      - example.com
      - local

  db:
    image: postgres
    networks:
      - backend
    hostname: postgres-db

networks:
  backend:
    driver: bridge

# 容器內 /etc/resolv.conf 將被自動設定
# search example.com local
# nameserver 8.8.8.8
# nameserver 1.1.1.1
```

**Docker 守護程序級別設定：**

```json
{
  "dns": ["8.8.8.8", "1.1.1.1"],
  "dns-search": ["example.com"],
  "insecure-registries": [],
  "registry-mirrors": ["https://mirror.example.com"]
}
```

#### 自定義服務發現（Service Discovery）

**使用 Docker 內建 DNS 的服務發現：**

```bash
# 建立自定義網路
docker network create mynet

# 執行服務
docker run -d --name web --network mynet nginx:latest
docker run -d --name db --network mynet postgres:latest

# 在其他容器中透過服務名訪問
docker run -it --network mynet busybox sh
# ping web  # 自動解析到 web 容器 IP
# ping db   # 自動解析到 db 容器 IP
```

**Compose 服務名自動發現：**

```yaml
version: '3.9'

services:
  frontend:
    image: nginx
    depends_on:
      - backend
    environment:
      BACKEND_URL: http://backend:8080

  backend:
    image: myapp
    depends_on:
      - database

  database:
    image: postgres
    environment:
      POSTGRES_DB: mydb

# frontend 容器可以直接訪問 http://backend:8080
# backend 容器可以直接訪問 postgres://database:5432
```

#### DNS 效能最佳化

```bash
# 檢查 DNS 延遲
time docker exec <container> nslookup www.example.com

# 最佳化 DNS 解析
docker run -d \
  --dns 127.0.0.1 \  # 使用本地快取 DNS (Dnsmasq)
  nginx:latest

# 在 Kubernetes 中最佳化
kubectl patch deployment -n kube-system coredns --patch '{
  "spec": {
    "template": {
      "spec": {
        "containers": [
          {
            "name": "coredns",
            "resources": {
              "limits": {
                "memory": "512Mi",
                "cpu": "500m"
              }
            }
          }
        ]
      }
    }
  }
}'
```

### 9.7.4 網路策略（NetworkPolicy）實踐

網路策略定義了容器間的流量控制規則，是微服務架構中的安全基礎。

#### 基本網路策略

**預設拒絕所有入站流量的策略：**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: default
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  # 不指定 ingress 規則，表示拒絕所有入站流量
```

**允許特定來源的入站流量：**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-frontend
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: frontend
    ports:
    - protocol: TCP
      port: 8080
```

**允許出站流量到資料庫：**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-to-db
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
  - Egress
  egress:
  # 允許到資料庫的流量
  - to:
    - podSelector:
        matchLabels:
          tier: database
    ports:
    - protocol: TCP
      port: 5432
  # 允許 DNS 查詢
  - to:
    - namespaceSelector: {}
    ports:
    - protocol: UDP
      port: 53
  # 允許到外部 API 的流量
  - to:
    - ipBlock:
        cidr: 0.0.0.0/0
        except:
        - 169.254.169.254/32  # 阻止元資料服務
    ports:
    - protocol: TCP
      port: 443
```

#### 微服務網路策略範例

```yaml
---
# 拒絕所有預設
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

---
# Frontend 容器策略
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  # 允許來自 Ingress Controller 的流量
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx
    - podSelector:
        matchLabels:
          app: ingress-controller
    ports:
    - protocol: TCP
      port: 3000
  egress:
  # 允許到 API 的流量
  - to:
    - podSelector:
        matchLabels:
          app: api
    ports:
    - protocol: TCP
      port: 8080
  # 允許 DNS
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53

---
# API 容器策略
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  # 允許到資料庫的流量
  - to:
    - podSelector:
        matchLabels:
          app: postgres
    ports:
    - protocol: TCP
      port: 5432
  # 允許 DNS
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53

---
# 資料庫容器策略
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-postgres
spec:
  podSelector:
    matchLabels:
      app: postgres
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api
    ports:
    - protocol: TCP
      port: 5432
```

#### 使用 Calico/Cilium 的高階網路策略

**L7 應用層策略（僅 Cilium 支援）：**

```yaml
apiVersion: "cilium.io/v2"
kind: CiliumNetworkPolicy
metadata:
  name: "api-gateway-policy"
spec:
  description: "L7 policy for API gateway"
  endpointSelector:
    matchLabels:
      app: api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        # 允許 GET /api/users
        - method: "GET"
          path: "/api/users/.*"
        # 允許 POST /api/users 僅從管理員來源
        - method: "POST"
          path: "/api/users"
          sourceIPs:
          - "10.0.0.0/8"
```

### 9.7.5 跨主機容器通訊方案對比

#### 方案對比表

| 方案 | 隔離性 | 效能 | 複雜度 | 適用場景 |
|------|--------|------|--------|---------|
| 主機網路 | ✗ | ⭐⭐⭐⭐⭐ | 低 | 高效能，單主機 |
| Bridge + Host Port | 中 | ⭐⭐⭐⭐ | 低 | 小規模叢集 |
| Overlay (VXLAN) | ✓ | ⭐⭐⭐ | 中 | 跨域通訊 |
| BGP (Calico) | ✓ | ⭐⭐⭐⭐ | 中 | 大規模叢集 |
| eBPF (Cilium) | ✓ | ⭐⭐⭐⭐⭐ | 高 | 高效能大叢集 |

#### 選擇建議

```bash
# 1. 開發環境：使用 Bridge 網路
docker network create my-app
docker-compose up  # 預設使用 bridge

# 2. 小規模生產（< 50 節點）：使用 Flannel
kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml

# 3. 中等規模（50-500 節點）：使用 Calico
kubeadm init --pod-network-cidr=192.168.0.0/16
kubectl apply -f https://docs.projectcalico.org/v3.24/manifests/tigera-operator.yaml

# 4. 大規模（> 500 節點）或需要 L7 策略：使用 Cilium
helm install cilium cilium/cilium --namespace kube-system

# 5. 需要多雲/跨域：使用 Weave
```

### 9.7.6 網路故障排查

**常見網路問題診斷：**

```bash
# 1. 容器無法訪問外部網路
docker exec <container> ping 8.8.8.8
docker exec <container> cat /etc/resolv.conf
docker logs <container> | grep -i network

# 2. 容器間無法通訊
docker network inspect <network>
docker exec <container1> ping <container2>

# 3. 連接埠對映失效
docker port <container>
netstat -tlnp | grep <port>

# 4. DNS 解析失敗
docker exec <container> nslookup example.com
docker exec <container> cat /etc/hosts

# 5. 網路延遲
docker run --rm --network host iperf3:latest -c <target>
docker exec <container> mtr -r example.com

# 使用 tcpdump 抓套件分析
docker run --rm --cap-add NET_ADMIN --network host \
  corfr/tcpdump -i eth0 -n "port 80"
```
